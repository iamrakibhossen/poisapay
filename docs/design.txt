1. Introduction

1.1 Purpose
Define the technical design of PoisaPay: a custodial wallet that lets users in the Bangladesh market hold, receive,
send, and (later) exchange crypto assets across Ethereum, Tron, and BNB Smart Chain, with all keys held and
operated by the platform. The document exists so that engineering, security, and compliance can review and
agree the design before large-scale implementation, and so that each subsystem has an unambiguous target.

1.2 Scope
In scope: custody (HD + hot/cold + multisig), a double-entry ledger, deposit/withdrawal/sweep money
movement, blockchain monitoring, internal transfers, merchant payments and QR, exchange/swap, KYC/AML/risk,
admin operations, REST API + webhooks, and the infrastructure to run and scale it.
Out of scope (this version): native mobile apps, card issuance, fiat on/off-ramp mechanics, and multi-currency
beyond the launch assets — all tracked as future work (§13, Phase 10).

2. System Overview

2.1 Product vision
PoisaPay is a custodial, multi-chain wallet for the Bangladesh market. The custodial choice is deliberate: it lets
ordinary users transact without managing keys, at the cost of the platform bearing full custody responsibility.
That single decision drives the entire design toward two obsessions — an auditable ledger and isolated key
custody.

3.2 Trust zones (the security spine)
Three isolation zones, and the design's most important boundary is the gap between the first two:
1. Online zone — web/API, admin, Address Generator, Blockchain Monitor. Holds no private keys. A full breach
   here lets an attacker read balances and generate addresses, not move funds.
2. Signing zone — Sweep & Withdrawal signers. Network-isolated, holds hot keys (decrypted only in memory
   per operation), amount-capped and rate-limited with a circuit breaker. Blast radius = the small hot float.
3. Cold zone — offline. Master seed + cold multisig keys. Never online; cold→hot is human-gated M-of-N.

4. Custody Design

4.1 HD wallet hierarchy
One BIP39 seed (24 words, generated air-gapped) derives everything. Account level uses hardened derivation
per chain; address level uses non-hardened so deposit addresses can be derived from an account xpub without
private keys (see Volume I §8).

 Chain                      Path template                            Coin type      Encoding

 Ethereum                    m/44'/60'/0'/0/{i}                      60             EIP-55 ( 0x… )

 BNB Smart Chain             m/44'/60'/1'/0/{i}                      60 (EVM)       EIP-55 ( 0x… )

 Tron                        m/44'/195'/0'/0/{i}                     195            Base58Check ( T… )


4.2 Address generation
The Address Generator receives only the account xpub and derives /0/{index} via public derivation. It cannot
produce a private key. The derivation index is allocated atomically from a per-chain monotonic counter under a
row lock (no MAX+1 ). See Volume I §8.3–8.5.

4.3 Hot / Cold / Multisig
 • Hot wallet: minimal operational float (floor/target/cap policy). Funds withdrawals and sweep gas. Its balance
   is the maximum online blast radius.
 • Cold wallet: offline bulk custody, watch-only monitored via xpub. Cold→hot is manual + M-of-N + air-gapped.
 • Multisig: EVM Safe-style contract; Tron native weighted-permission multisig. Proposal → M approvals →
   broadcast, with per-signer no-double-approve. See Volume II §16–18.

4.4 Key ceremony
Seed generated air-gapped, split via Shamir across custodians, xpubs registered online, hot keys provisioned into
the secrets vault. Executed at delivery Phase 4 with a signed, archived runbook.



5. Ledger Design

5.1 Double-entry model
Every value movement is a balanced journal entry of ≥2 lines where Σ debits = Σ credits , enforced in code
and by a deferred DB trigger. Accounts include user wallets and system accounts ( treasury:hot|cold|pending ,
fee:income , gas:expense , liability:user-funds ). See Volume I §9.

5.2 Balance components
Each user wallet exposes available , locked (reserved), and total = available + locked . Locking is itself a
balanced transfer user:available → user:locked — no special-case field mutation.

5.3 States & idempotency
Business events move pending → completed | failed | reversed . Lines are never edited; a correction is a new
reversing entry linked by reverses_entry_id . Every entry carries an idempotency_key (e.g. deposit:{txhash}:
{vout} ); a retry collapses to a no-op.

5.4 Reconciliation
Scheduled per asset: assert on-chain controlled ≈ ledger treasury (within an in-flight tolerance) and ledger
treasury ≥ ledger liability (solvency). Per-address completeness checks recover missed deposits after an
outage. Drift or insolvency pages on-call and can freeze operations. See Volume II §12.9.

6. Money Movement Design (step-by-step flows)

6.1 Deposit flow

  Step 1   User requests a deposit address (or reuses one) for asset-on-chain.
  Step 2   API allocates index (atomic) → Address Generator derives address → store + add to Redis watch set.
  Step 3   User sends funds on-chain to the address.
  Step 4   Monitor detects the tx (native scan or token Transfer log); records onchain_tx + deposit (detected).
  Step 5   On each new block, confirmations recomputed; on reorg, canonical re-check.
  Step 6   At min_confirmations + still-canonical → post entry (treasury:pending → user:available). Deposit=credited.
  Step 7   Reverb pushes the new balance to the user in realtime.
  Step 8   (Async, later) Sweep policy may move the coins to the hot wallet — independent of the user's balance.



6.2 Sweep flow

  Step 1   Deposit-address token balance crosses per-asset threshold (or scheduled).
  Step 2   Fund gas: EVM → gas-station sends native coin to the address; Tron → delegate energy/bandwidth.
  Step 3   Sweep Signer builds + signs the token-move tx (nonce-managed on EVM; expiry-managed on Tron).
  Step 4   Broadcast (idempotent by nonce/context); monitor confirms.
  Step 5   Ledger posts sweep.settle (treasury:hot → treasury:pending) + gas expense. Sweep=swept.


Note: sweep never touches the user's wallet; it is treasury-internal.

6.3 Withdrawal flow

  Step 1   User submits withdrawal (asset, amount, to_address); validated (min/max, address format).
  Step 2   Risk/AML checks run (screening, velocity). Above auto-threshold or flagged → manual admin approval.
  Step 3   RESERVE FIRST: lock funds (user:available → user:locked). Balance can't be double-spent now.
  Step 4   Withdrawal Signer signs + broadcasts from hot wallet (nonce-managed, idempotent by id).
  Step 5   Monitor confirms at N confs.
  Step 6   Ledger settles: debit user:locked, credit treasury:out, debit fee. Withdrawal=completed.
  Step 7   If cancelled/failed BEFORE broadcast → reverse the lock (user:locked → user:available).


Invariant: reserve before you sign — never touch the chain before the ledger has locked the funds.

6.4 Internal transfer
User→user transfer is a pure ledger operation (no chain): a single balanced entry sender:available →
recipient:available , atomic, idempotent, instantaneous, zero network fee. This is a major UX and cost
advantage of the custodial model.

6.5 Chain-specific notes
 • ERC-20 / BEP-20: detect via Transfer event logs (contract + topic0), match to in-app against the Redis set.
   EVM nonce + EIP-1559 gas.
 • TRC-20: detect via Tron event logs; hex↔base58 conversion; solidity-block confirmations; energy delegation
   for cheap sweeps.
 • Respect per-asset decimals (USDT=6, native=18); store base units verbatim.

On-chain: NUMERIC(78,0) base units. Ledger: NUMERIC(38,0) minor units. No float/double anywhere. Display
formatting applies decimals only at the edge.

9. Security Design

9.1 Trust boundaries
Per §3.2: online (no keys) / signing (isolated, capped) / cold (offline, multisig). This is the primary control.

9.2 Authentication & authorisation
2FA (TOTP), OTP (SMS/email, daily-capped), device verification, server-side session management with revocation,
and RBAC/permission system for both users and operators (Spatie-style). Admin actions are permission-gated and
audit-logged.

9.3 Encryption & secrets
TLS in transit (Cloudflare + origin). Hot keys encrypted at rest via envelope encryption (data key wrapped by
KMS/HSM), decrypted only in signer memory per operation. Secrets (RPC keys, KMS grants, DB creds) in a vault,
injected at runtime, never in repo or image. DB CHECK s refuse anything key-shaped in address/hot tables.

9.4 Signer controls
Amount caps + rate limits at the infrastructure level; a circuit breaker trips if rolling outflow exceeds the
expected envelope, freezing signing and paging on-call. Even a fully compromised signer cannot exceed the
small hot float before a human is alerted.

9.5 Rate limiting, fraud, audit
Dual-window rate limiting (per-account + per-IP) with CAPTCHA escalation on auth; withdrawal risk scoring and
daily outflow circuit breakers; immutable, tamper-evident audit trail for all privileged and money-moving actions.
Full treatment in Volume V (planned).



10. Compliance & Risk Design

10.1 KYC (tiered)
Identity verification (NID/passport + liveness/document check via provider) with verification tiers that gate
limits: unverified (view-only / no withdrawal), basic, and full (higher limits). Onboarding cannot progress to real-
money actions without the required tier.

10.2 AML
Sanctions/PEP screening at onboarding and per-withdrawal; transaction monitoring with velocity/pattern rules; a
SAR (suspicious activity report) workflow for compliance operators. Screening results are recorded and auditable.

10.3 Risk engine
Per-withdrawal risk scoring feeding the auto-approve vs. manual-review decision; per-user and platform-wide
daily outflow circuit breakers; anomaly flags surfaced to operators. Thresholds are configurable data.

10.4 Bangladesh regulatory posture (design input)
The regulatory status of custodial crypto services in Bangladesh is restrictive and subject to change. This TDD
treats compliance as a hard gate (no real-money onboarding before KYC/AML is live — §13) and isolates
compliance so the posture can be tightened without re-architecting. Specific licensing, permitted-activity, and
reporting obligations are a matter for qualified legal counsel and are an explicit dependency, not an engineering
assumption.



11. Infrastructure & Deployment Design

11.1 Container topology
Docker services orchestrated by Coolify: app (PHP-FPM), nginx , postgres (+ replicas), redis-cache ,
redis-queue , horizon workers (multiple, by queue), reverb , and isolated signer services on separate hosts/
networks. Multi-stage images, pinned versions, opcache.validate_timestamps=0 , config/route/view cache on
deploy.

11.2 Environments
local (Docker), staging (testnet keys, production-like), production (mainnet). Testnet-first: every chain flow is
proven on staging/testnet before mainnet keys exist.

11.3 Networking & zone isolation
Cloudflare (WAF/DDoS/TLS) → Nginx → app. Signer hosts have no inbound internet; they pull signing jobs and
push signed txs. Cold zone is fully offline. Wildcard TLS via DNS-01 for tenant/subdomain needs (per prior Caddy/
Cloudflare experience; Nginx used here per stack).

11.4 Scaling
PostgreSQL read replicas + PgBouncer (transaction pooling); Redis split (cache allkeys-lru ; queue/session
noeviction + AOF); Horizon per-queue concurrency so deposit-crediting, sweeping, and withdrawal-signing never

Centralised logs/metrics/traces; alerting on reconciliation drift, signer circuit breaker, queue depth, provider

with runbooks.

F1. Multi-Currency & Local (Fiat) Accounts

F1.1 Design
Generalise the asset model so the ledger holds fiat currencies (BDT, USD, EUR…) exactly as it holds crypto. A
fiat "account" is just a ledger_account whose asset is a currency with decimals = 2 . Everything the ledger
already guarantees — double-entry, idempotency, available/locked — applies unchanged.
 • Base display currency: the app shows a user's total in one chosen currency (e.g. BDT or USD), computed
   from live FX; the stored balances remain per-asset and exact.
 • Payment priority: the user orders which assets are spent first (e.g. USDT → BDT → BTC). Stablecoin-first is
   recommended to minimise volatility exposure.

F1.3 On-ramp flow (fund fiat balance)

  Step 1   User initiates on-ramp: amount + rail (bank / mobile wallet).
  Step 2   Create ramp_order (pending). Present pay instructions / redirect to PSP.
  Step 3   User pays via the rail; PSP sends a confirmation callback (HMAC-verified, idempotent by provider_ref).
  Step 4   On confirmed funds → post ledger entry (ramp:clearing -> user:available:BDT). ramp_order=credited.
  Step 5   User's BDT balance updates in realtime (Reverb).


Off-ramp is the mirror: lock fiat → PSP payout to the user's bank/wallet → on payout confirmation, settle.

  ⚠ On/off-ramp settlement is asynchronous and partner-paced; never credit before the PSP confirms cleared
  funds (same discipline as on-chain confirmations).




F2. Exchange & Conversion Engine

F2.1 Design
One engine serves three needs: crypto↔crypto swaps, crypto↔fiat conversion, and the just-in-time (JIT)
conversion that card spending depends on (F3). It is quote-driven with an explicit spread:
 • Quote: given (from_asset, to_asset, amount) , return a rate (from a rate source), a spread (bps), the
   resulting amount, and an expiry (quotes are short-lived — seconds — because rates move).
 • Execute: re-validate the quote hasn't expired, lock the from amount, post a balanced conversion entry,
   credit the to amount. Spread accrues to fx:spread_income .

F2.3 Volatility / FX risk
JIT conversion means PoisaPay carries price risk between a card authorisation hold and its settlement. Mitigations
designed in: prefer stablecoins in spending priority; hold a small buffer above the quoted amount; and run a
treasury hedging/spread policy so the aggregate spread income covers realised slippage. This is a treasury-
operations concern (Volume II family), surfaced here because cards create the exposure.



F3. Card System (the "card generator")
The headline feature. PoisaPay does not issue cards itself — it integrates a licensed issuer / BIN-sponsor +
processor and a card network (Visa/Mastercard). "Card generator" = provision a virtual/physical card via the
partner API, then authorise each spend in real time against the user's balances with JIT conversion.

F3.1 PCI scope — stay out of it by design
PoisaPay never stores or touches the PAN/CVV. It holds only a network token / issuer card reference. Card
reveal and Apple/Google/Samsung Pay provisioning happen through the partner's PCI-compliant vault (iframe/
SDK), so PoisaPay's own systems stay out of PCI-DSS cardholder-data scope. This is the single most important
card-security decision.

F3.2 Card lifecycle
Issue (virtual instant / physical shipped) → active → (freeze/unfreeze, set limits, rotate) → spend → (dispute/
refund) → close. Cards require a higher KYC tier (§10.1). Types: virtual (online + mobile-wallet) and physical
(POS + ATM + tap-to-pay).

F3.3 The authorisation flow (latency-critical)
This is the crown-jewel sequence. The card network gives only a few seconds to approve, so the auth path must
be fast — Redis-cached balances and pre-warmed quotes, no slow queries.

  Step 1   User taps/swipes; network → issuer-processor → real-time AUTH webhook to PoisaPay
           (card token, amount, settlement currency, MCC, merchant).
  Step 2   Resolve card -> user; check card status, KYC tier, per-card + velocity limits, risk score.
  Step 3   Pick funding asset by the user's spending priority; get a JIT crypto->fiat quote (incl. spread).
  Step 4   Sufficient available balance? If not (or risk/limit fails) -> DECLINE (respond fast).
  Step 5   Place HOLD: ledger locks the crypto equivalent (user:available -> user:card_hold).
  Step 6   Respond APPROVE to the webhook within the network timeout. (p99 budget: ~2s — hard NFR.)
  Step 7   Later, network sends CLEARING/SETTLEMENT (amount may differ: tips, partial, FX).
  Step 8   Settle: convert held crypto->fiat (realise via F2), debit user:card_hold,
           credit card_program:settlement, post card fee + fx spread. Release any over-hold.
  Step 9   Refund: merchant refund -> credit back to user balance per policy. Dispute -> F3.6 workflow.

F3.6 Disputes & chargebacks
Card programs carry chargeback liability. A dispute reverses the settlement in the ledger (a reversing entry,
never a mutation), tracks representment with the issuer, and records win/loss. Chargeback losses hit a
card_program:loss account and are monitored — high dispute rates are both a cost and a risk signal.

F4. Send Money, Transfers & Payouts
Three distinct products, one ledger:

F4.1 Internal P2P (instant, free)
Send to another PoisaPay user by handle (phone / email / username). Pure ledger op: one balanced entry
 sender:available -> recipient:available , atomic, idempotent, instant, zero network fee. Supports crypto and
fiat assets. If the recipient handle isn't registered, hold as a claimable pending transfer with expiry.

F4.2 Fiat payout (off-platform)
Send fiat to an external bank account or mobile wallet — an off-ramp targeted at a third party. Lock funds → PSP
payout → settle on confirmation. Same machinery as F1 off-ramp with an external beneficiary.

F4.3 Remittance (cross-border) — high value for the Bangladesh market
Cross-border remittance is a major use case here. Sender funds in crypto/fiat; recipient receives BDT to a local
bank or mobile wallet. Designed as: source lock → cross-border settlement (via stablecoin rail internally) → local
BDT payout via PSP. Compliance-heavy (both ends KYC'd, AML screened, purpose codes) — gated behind the
money gate and remittance-specific permissions.

F5. Rewards & Referral

F5.1 Design
Rewards are ledger credits into rewards:* accounts, claimable/spendable per policy. Three mechanisms:
welcome bonus, referral rewards (referrer + referee), and optional card cashback / daily stablecoin rewards.
Keeping rewards in the ledger means they reconcile like any other balance and can't be double-granted
(idempotency on the grant).

F5.2 Schema

 CREATE TABLE referrals (
     id           UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
     referrer_id UUID         NOT NULL REFERENCES users(id),
     referee_id   UUID        NOT NULL REFERENCES users(id),
     code         VARCHAR(24) NOT NULL,
     status       VARCHAR(16) NOT NULL DEFAULT 'pending', -- pending|qualified|rewarded

Rewards grants are idempotent ( reward:{type}:{user}:{period} ) and anti-abuse rules gate qualification (referee
must complete KYC + a qualifying action) to prevent farming.



F6. Crypto-Backed Credit (design note)
A borrow-against-collateral product: the user pledges crypto as collateral and draws a credit line (in fiat or
stablecoin) without selling, repaying flexibly. Core mechanics: collateral lock, loan-to-value (LTV) ratio, interest/fee
accrual, margin monitoring, and liquidation if collateral value falls below a maintenance threshold.
 • Ledger: user:collateral:locked , credit:principal , credit:accrued_fee , and a liquidation path that sells
   collateral (via F2) to cover the outstanding line.
 • Risk: continuous mark-to-market of collateral; automated margin calls and liquidation; conservative LTV.

  ⚠ Lending is separately regulated and introduces liquidation/market-risk machinery beyond the wallet core.
  This is a later-phase product, gated on lending permission and robust market access — designed here for
  completeness, not for near-term build.
